# AI生成コード レビュー

AI が書いたコードは、一見すると正しく動く。テストも通る。しかし、AI には人間のエンジニアが自然に持っている3つの能力が欠けている：

1. **システム全体を見渡す力** — AI は与えられたタスクの範囲で最適化する。隣のモジュールへの影響、既存の設計パターンとの一貫性、将来の拡張性は考慮の外にある。
2. **「なぜ」を問う力** — AI は「それっぽい」コードを生成する。なぜその設計を選んだのか、なぜその名前にしたのか、という意図は持たない。
3. **プロジェクト固有の文脈** — AI は汎用パターンを再現する。このプロジェクトの歴史、チームの慣習、運用上の制約は知らない。

このスキルは、これらの盲点を人間の視点で埋めるレビュープロセスを提供する。

## レビュー対象の特定

レビュー対象は以下のいずれかの方法で特定する：

- ユーザーが指定したファイルやディレクトリ
- `git diff` で検出された変更差分（ステージ済み・未ステージ両方）
- PR番号が指定された場合は `gh pr diff` で取得
- 直前のAIによるコード変更

対象が不明確な場合はユーザーに確認する。

## レビュープロセス

### Phase 1: 変更の全体像を掴む

レビューに入る前に、まず変更の全体像を理解する。

1. **変更差分を取得する** — `git diff` や対象ファイルを読む
2. **変更の目的を確認する** — コミットメッセージ、PR説明、またはユーザーへの質問から「何を達成しようとしているか」を把握する
3. **影響範囲をマッピングする** — 変更されたファイルが依存している／されているモジュールを特定する

この段階で変更の目的が不明確な場合は、ユーザーに「この変更で何を実現したいか」を確認する。目的がわからないままレビューしても、本質的な問題は見つけられない。

### Phase 2: 3つの観点でレビュー

`references/checklist.md` を読み、以下の3つの観点で検証する。各観点はAIの構造的弱点に直接対応している。

#### 観点1: システム整合性（← 局所最適への対抗）

AI は目の前のタスクを解決するが、変更がシステム全体にどう影響するかは考えない。この観点では「木を見て森を見ず」になっていないかを検証する。

確認すること：
- **既存コードとの一貫性** — 同じことを別の方法でやっていないか。既存のユーティリティやパターンが無視されていないか
- **波及影響** — この変更が他のモジュールの動作や契約を壊さないか
- **重複と矛盾** — 既存の仕組みと重複する実装が作られていないか。既存の設計方針と矛盾していないか
- **境界の整合性** — API の入出力型、DB スキーマ、設定値などのインターフェースが既存の契約と整合しているか

#### 観点2: 意図と意味（← 意図の欠如への対抗）

AI は統計的に「ありそうな」コードを生成する。だが「なぜこの実装なのか」という根拠は持たない。この観点では、コードの意図が明確で、ビジネス要件に正しく対応しているかを検証する。

確認すること：
- **命名の的確さ** — 変数名・関数名・クラス名が実際の役割を正確に表現しているか。曖昧な名前（`data`, `handler`, `process`, `utils`）で意図が隠れていないか
- **ビジネスロジックの正確さ** — 実装がビジネス要件を正しく反映しているか。「それっぽいが微妙に違う」実装になっていないか
- **エラーハンドリングの意味** — エラー処理が形式的でなく、実際の障害シナリオに対応しているか。握りつぶし（silent catch）や汎用すぎる例外処理がないか
- **コメントと実装の一致** — コメントが実装を正確に説明しているか。AI が生成した「それっぽいが不正確な」コメントがないか

#### 観点3: プロジェクト適合性（← 学習ベースの限界への対抗）

AI は Stack Overflow や GitHub で学んだ汎用パターンを再現する。だが、目の前のプロジェクトが持つ固有の文脈 — 技術スタック、チーム規約、運用環境、過去の経緯 — は知らない。

確認すること：
- **プロジェクト規約への準拠** — コーディングスタイル、ディレクトリ構成、命名規則がプロジェクトの慣習に従っているか
- **技術スタックとの整合** — プロジェクトで使っていないライブラリや、既に採用されているライブラリの別バージョンを持ち込んでいないか
- **非機能要件** — パフォーマンス、セキュリティ、アクセシビリティなど、プロジェクト固有の品質基準を満たしているか
- **運用への配慮** — ロギング、モニタリング、デプロイ、ロールバックなど、運用に必要な考慮が抜けていないか

### Phase 3: 発見事項の整理と報告

## レポートフォーマット

レビュー結果は以下の構造で報告する。発見事項がない観点は省略してよい。

```
## レビュー対象
（変更の範囲と目的を1-2文で記載）

## 発見事項

### [致命的 / 重要 / 改善提案] ファイル名:行番号 — 要約
- **観点**: システム整合性 / 意図と意味 / プロジェクト適合性
- **問題**: 具体的に何が問題か
- **理由**: なぜそれが問題なのか（AIの盲点との関連を含む）
- **修正案**: 具体的にどう直すべきか

（発見事項ごとに繰り返す）

## 総合判断
- ✅ マージ可 — 問題なし、または軽微な改善提案のみ
- ⚠️ 要修正 — 重要な問題あり、修正後に再レビュー推奨
- 🚫 差し戻し — 致命的な問題あり、設計の見直しが必要
```

### 重要度の基準

- **致命的**: データ損失、セキュリティ脆弱性、本番障害に直結する問題
- **重要**: 正しく動作するが、保守性・信頼性・一貫性に深刻な影響がある問題
- **改善提案**: 現時点で害はないが、改善すればコードの質が上がる提案

## レビューの姿勢

- **コードを読む前に目的を理解する** — 目的を知らずにコードを読むと、「動くかどうか」しか見えない。「正しいことをしているか」は目的がわかって初めて判断できる。
- **AI の盲点を意識する** — 「正しく見えるが実は違う」パターンに注意する。AI のコードは表面的には整っていることが多いため、構文的な問題よりも設計や意味の問題に集中する。
- **修正案は具体的に** — 「改善が必要」ではなく「この行をこう変える」まで踏み込む。レビューを受けた側（AI であれ人間であれ）がすぐに行動できる指摘にする。
- **過剰なレビューを避ける** — 全てを指摘する必要はない。致命的・重要な問題に集中し、些末な指摘でノイズを増やさない。
