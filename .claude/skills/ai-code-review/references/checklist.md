# AI生成コード レビューチェックリスト

AI の構造的弱点に対応した具体的なチェック項目。レビュー時にこのリストを参照し、該当する項目を検証する。全項目を機械的にチェックするのではなく、変更の内容に関連する項目に絞って深く検証すること。

---

## 観点1: システム整合性（← 局所最適への対抗）

AI は「このタスク」の中で最適なコードを書くが、「このシステム」の中で最適かは考えない。

### 一貫性

- [ ] 既存コードで同じ問題を解決しているパターンがある場合、それと一貫した方法を使っているか
- [ ] 新しいユーティリティ関数やヘルパーを追加する前に、既存のものを探したか
- [ ] エラーハンドリングのパターンがプロジェクト内で統一されているか（例: Result型 vs 例外 vs エラーコード）
- [ ] ログ出力のフォーマットや粒度が既存のコードと揃っているか

### 波及影響

- [ ] 共通インターフェース（API、DB スキーマ、共有型）を変更する場合、全ての利用箇所を確認したか
- [ ] 依存パッケージのバージョン変更が他のモジュールに影響しないか
- [ ] 設定値やフィーチャーフラグの変更が他の環境（staging、production）に影響しないか
- [ ] 既存のテストが意図せず壊れていないか

### 重複

- [ ] 既存の関数・クラス・モジュールと実質的に同じ機能を持つコードが新たに作られていないか
- [ ] コピペで持ってきたような似たコードブロックがないか
- [ ] 既に deprecate 予定のパターンを新たに使っていないか

---

## 観点2: 意図と意味（← 意図の欠如への対抗）

AI は「ありそうな」コードを生成する。人間は「正しい」コードを書く。その差は「意図」の有無にある。

### 命名

- [ ] 関数名が「何をするか」を正確に表現しているか（`processData` → `validateAndSaveUserProfile` のように）
- [ ] 変数名が中身を正確に表しているか（`tmp`, `data`, `result` のような曖昧な名前がないか）
- [ ] boolean 変数が疑問文になっているか（`isValid`, `hasPermission`, `canRetry`）
- [ ] 略語が一般的でないものを避けているか

### ビジネスロジック

- [ ] 要件の境界条件が正しく実装されているか（「以上」と「超過」、「含む」と「含まない」）
- [ ] 金額・日付・時刻などの精度が要件と一致しているか
- [ ] ステート遷移が仕様通りか（許可されていない遷移がブロックされているか）
- [ ] 排他制御が必要な箇所で適切に実装されているか

### エラーハンドリング

- [ ] catch ブロックが例外を握りつぶしていないか（catch (e) {} は危険信号）
- [ ] エラーメッセージが具体的で、デバッグに役立つ情報を含んでいるか
- [ ] リトライロジックがある場合、最大回数やバックオフが適切か
- [ ] 外部サービス呼び出しにタイムアウトが設定されているか
- [ ] 部分的な失敗時のロールバックやクリーンアップが考慮されているか

### コメント

- [ ] コメントが「何をしているか」ではなく「なぜそうしているか」を説明しているか
- [ ] AI が生成した不正確なコメントがないか（実装と矛盾するコメント）
- [ ] TODO/FIXME/HACK が放置されていないか
- [ ] 自明なコードに冗長なコメントがついていないか

---

## 観点3: プロジェクト適合性（← 学習ベースの限界への対抗）

AI は「一般的なベストプラクティス」を知っているが、「このプロジェクトのベストプラクティス」は知らない。

### 規約

- [ ] ファイルの配置がプロジェクトのディレクトリ構成ルールに従っているか
- [ ] import/export のスタイルがプロジェクト内で統一されているか
- [ ] テストファイルの命名と配置がプロジェクトの慣習に従っているか

### 技術スタック

- [ ] プロジェクトで使っていないライブラリを新たに追加していないか（既存ライブラリで代替可能な場合）
- [ ] 使用しているAPIやメソッドがプロジェクトのターゲットバージョンで利用可能か
- [ ] ORM やデータアクセス層の使い方がプロジェクトの方針に従っているか

### セキュリティ

- [ ] ユーザー入力が適切にバリデーション・サニタイズされているか
- [ ] SQLインジェクション、XSS、CSRF などの一般的な脆弱性がないか
- [ ] 秘密情報（APIキー、パスワード）がハードコードされていないか
- [ ] 認証・認可チェックが適切な箇所で行われているか
- [ ] ログに個人情報や秘密情報が出力されていないか

### パフォーマンス

- [ ] N+1 クエリなどの明らかな非効率がないか
- [ ] 不必要なデータの全件取得がないか（ページネーション、フィルタリング）
- [ ] メモリリークにつながるパターンがないか（イベントリスナーの解除漏れなど）
- [ ] ホットパスでの不要な処理（重い計算、I/O）がないか

### 運用

- [ ] 適切なログ出力があるか（多すぎず少なすぎず）
- [ ] 障害時の調査に必要な情報がログに含まれているか
- [ ] マイグレーションがある場合、ロールバック可能か
- [ ] フィーチャーフラグやグレースフルデグラデーションが必要な変更ではないか
